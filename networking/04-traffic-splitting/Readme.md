In this section we will learn in detail about how to use Cilium to perform canary deployment by splitting traffic between two applications or two version of applications

# Practicals
## 1. Deploying workload
```bash
# Create namespace
kubectl create ns app-space

# Deploying app version 1 with svc (see the manifest for details)
kubectl apply -f app-v1.yaml -n app-space

# Deploying app version 2 with svc (see the manifest for details)
kubectl apply -f app-v2.yaml -n app-space
```

### Knowledge base [Need for two separate service]
To answer the architectural question regarding why we need two distinct Kubernetes Services: in a traditional setup, a single Service uses a label selector (like app: my-app) to group all pods together. If you want a 90/10 split there, you would have to manage the ratio of pods manually (e.g., 9 v1 pods and 1 v2 pod).

By using two separate services (svc-v1 and svc-v2), we completely separate the application scaling from the traffic routing. The Gateway API handles the mathematical split at the network layer, letting Cilium's eBPF data path shift the traffic dynamically without touching your pods.

## 2. Canary Split (HTTP Route)
We will now create an HTTPRoute in the app-space namespace. This route will attach to your central infra-gateway (located in the infra-gateway namespace) and split the incoming HTTP traffic: 90% to svc-v1 and 10% to svc-v2.

```bash
kubectl apply -f canary-route.yaml
```

## 3. Testing
We can test the traffic distribution directly. Let's run a continuous loop of 20 requests to watch the eBPF load-balancing weights in action.
```bash
for i in {1..20}; do curl -s http://<GatewayIP>/; done
```

### Knowledge base [Why Traffic split is not even]
- Statistically, with a 90/10 split, you should see Version 1 🟢 dominating the output, with Version 2 🚀 popping up roughly 2 times out of the 20 requests.

- However, since this is a statistical split and not a strict round-robin counter, you might see 1, 2, or even 3 requests land on v2 in a short test run.

- To understand why this happens, we have to look at how Cilium processes this weight allocation under the hood. Cilium doesn't use standard iptables rules like legacy kube-proxy. Instead, it programs an eBPF (Extended Berkeley Packet Filter) maps table directly inside the Linux kernel data path.

- When a request hits your Gateway IP (Eg: 172.19.0.2:80), the eBPF program attaches to the network interface, looks up the destination in a hash map, and runs a weighted random selection algorithm directly in kernel space to choose the backend endpoint (svc-v1 backend pods vs. svc-v2 backend pods).

- Let's test your understanding of this mechanism: If we scale down app-v2 to 0 replicas while leaving the HTTPRoute weights at 90/10, what do you think will happen to the 10% of traffic that Cilium's eBPF program tries to steer toward svc-v2 : *All the traffic will be routed to svc1 as the end point is 0. eBPF maps updating dynamically! 🗺️ When replicas change, Cilium instantly updates its internal state without needing a reload.*

- Lets get in detail: If svc-v2 has 0 endpoints, the eBPF data path doesn't automatically route that 10% of traffic back to svc-v1. Instead, because the HTTPRoute explicitly tells Cilium to send 10% of traffic to svc-v2, the kernel will still attempt to steer that 10% to the empty service. Since there are no backing pods, those requests will fail, typically resulting in a 503 Service Unavailable HTTP error for roughly 1 out of every 10 requests. 🛑

- This is why, in a production canary pipeline, you must ensure health checks pass before shifting weights, or use an automated progressive delivery tool (like Argo Rollouts) that orchestrates this safely.

## 3.1 Testing Zero Replica
```bash
## Scale down v2 to 0
kubectl scale deployment app-v2 -n app-space --replicas=0

## Check the pod of app-v2 it should be 0
kubectl get all -n app-space

## Now run the same test
for i in {1..20}; do curl -s http://<GatewayIP>/; done

```
### The output Analysis

The error message **"upstream connect error or disconnect/reset before headers"** is a classic message generated by the Envoy proxy (which Cilium manages under the hood for Layer 7 traffic routing).

Here is exactly what that message breaks down to: Upstream: In networking terms, "upstream" refers to the backend targets where traffic is being sent—in this case, your svc-v2 pods. 🗺️

Connect Error / Connection Timeout: Envoy faithfully honored your HTTPRoute weights and attempted to route roughly 10% of the incoming requests to svc-v2. However, because you scaled app-v2 down to 0, there were no active pod IP addresses for Envoy to connect to. 🛑

Because Envoy could not establish a connection to any backend pod for that specific upstream route, it terminated the request and passed that error string back to your curl loop.

This confirms that the traffic split is actively working in the cluster network, even when a backend is empty.
